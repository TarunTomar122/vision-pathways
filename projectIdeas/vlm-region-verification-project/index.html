Region-Grounded Verification for VLM OCR and Document Answers

A deployment reliability project targeting stable-but-wrong VLM OCR answers: outputs that are consistent across views but not actually supported by local visual evidence.

0 TL;DR

VLMs are increasingly used as OCR/document engines, but generative decoding often produces answers that are plausible rather than visually verifiable.

The gap: stable-but-wrong consensus.
Cross-view risk controllers accept an answer when multiple views agree. But sometimes all views agree on the same wrong answer.

This project adds a local evidence contract: an answer is accepted only if it is supported by a specific region crop, OCR/layout evidence, or a crop-level VLM verifier.

2 Core hypothesis

Cross-view consistency is a useful risk signal, but not proof of correctness.

Hypothesis:
Stable-but-wrong OCR/document errors happen because global VLM generation optimizes semantic plausibility, while OCR/document extraction needs local geometric support. A region-grounded verifier should catch errors that cross-view consensus accepts.

If region verification does not help, that is still useful: it means these errors are not merely unsupported globally, but locally plausible or caused by visual ambiguity.

5 Research questions

  • RQ1 - Residual failure: how often does cross-view consensus accept stable-but-wrong OCR/document answers?
  • RQ2 - Local evidence: can region-level verification detect errors that global consistency misses?
  • RQ3 - Verifier design: which verifier works best under a small budget: crop OCR, crop VLM, string constraints, local perturbation, or latent probe?
  • RQ4 - Coverage-risk: how much coverage is sacrificed to reduce exposed catastrophic errors?

6 Proposed method

Baseline: GRC-style controller
  • Query a frozen VLM on multiple transformed views/crops.
  • Apply structural checks.
  • Accept if outputs are stable enough.
  • Abstain otherwise.
New layer: region-grounded verification
  1. Localize candidate evidence: OCR boxes, document layout boxes, text search, or VLM-predicted region.
  2. Crop and re-read: run specialist OCR or a crop-level VLM verifier.
  3. Check support: answer appears in expected region and remains stable under crop perturbations.
  4. Decide: accept, abstain, or acquire more evidence.

9 Scope & phases

Week 1

Reproduce baseline

Run Qwen/SmolVLM on synthetic OCR prompts and implement/fork a GRC-style consensus baseline.

Week 2

Stable-wrong testbed

Generate forms/receipts with degradations that induce plausible but wrong answers.

Weeks 3-4

Region verifier

Localize evidence, crop candidate regions, re-read with OCR/VLM, define support score.

Weeks 5-6

Risk evaluation

Compare confidence, cross-view consensus, region verification, OCR hybrid, and optional latent probe.

Week 7

Real validation

Validate on a small real subset from DocVQA/OCRBench/TextVQA.

Week 8

Write-up

Failure taxonomy, risk-coverage curves, verifier ablation, report/blog, clean repo.

10 Expected outcomes

  • Minimum viable result: cross-view consensus accepts a measurable class of stable-but-wrong OCR/document errors.
  • Strong result: region-grounded verification catches a meaningful fraction of those errors at acceptable coverage cost.
  • Very strong result: a small frozen-model verifier plus specialist OCR creates an auditable accept/abstain contract that beats confidence, self-consistency, and global cross-view agreement on catastrophic error reduction.

13 Open decisions

  • Primary scope: word/field-level OCR extraction vs document VQA answer verification.
  • Verifier stack: specialist OCR only, VLM crop verifier only, or hybrid.
  • Main model: Qwen2.5-VL-3B for realistic local VLM, SmolVLM2 for fast iteration.
Recommendation: start with field-level OCR extraction on synthetic forms.
Then validate on a small real document/OCR benchmark. Use a hybrid verifier: OCR boxes + crop OCR + VLM crop check.
