Reliability-Aware KV-Cache Reuse for Vision-Language Models

When a VLM sees an image it has (nearly) seen before, can we reuse its expensive cached computation instead of redoing it — and how similar is "similar enough" before the model silently starts getting things wrong?

0 TL;DR

VLMs turn one image into hundreds–thousands of "visual tokens," which blow up the KV cache and make serving slow and costly. People reuse/compress that cache to save money — but they only report aggregate accuracy, which hides which capability quietly broke.

  • Part 1 (safe result): measure the reliability of KV reuse/compression, sliced by capability (OCR, tables, spatial grounding) + instance-flip rate + hallucination rate.
  • Part 2 (stretch): build a similarity-gated reuser for near-duplicate images (docs / UI screenshots that differ slightly), borrowing the position-independent caching recipe from text-RAG.

1 The idea in one paragraph

Exact-match reuse already ships (vLLM, LMCache). Same-image smart reuse just landed (VLCache). Video-frame similarity reuse exists (InfiniPot-V, Sali-Cache). The open seam is (a) near-duplicate cross-request reuse for documents/UI — not temporal video frames — and (b) the capability-sliced reliability of any of it. This project lives at that intersection.

2 Why it matters

Motivating stat: UI-TARS-1.5-7B needs 76 GB of GPU memory for just 5 screenshots.
Screenshot/document VLMs are the worst case for KV blow-up — reuse/compression isn't optional, it's mandatory.
  • Cost: visual tokens dominate the KV cache; images inflate the prompt massively.
  • Real volume: document AI (invoices, forms, contracts), UI/computer-use agents, retail visual search — all send near-duplicate images at scale (same template, same screen, same product from a slightly different angle).
  • The blind spot: a reused/quantized model can drop ~1% on average while flipping ~20% of individual answers and destroying a specific capability. In documents, the one flipped digit is the whole point.

Why video methods don't transfer to docs/UI

Video methods compare patch (x,y) at frame t to patch (x,y) at t+1 — assuming a single ordered stream, smooth motion, and aligned patches. Docs/UI break all three: no stream (different users/requests), near-duplicates are reflowed (insert a row → everything shifts positions), and a tiny visual change can be a huge semantic change (a 0 → 8). So you need a content similarity gate + alignment, not an optical-flow temporal diff.

4 Research questions

  • RQ1 (reliability): When we reuse/compress VLM KV, which capabilities degrade first (OCR/text, tables, charts, spatial grounding), and how much does the aggregate number hide it?
  • RQ2 (gating): For near-duplicate images, what cheap signal (perceptual hash, CLIP/ViT-embedding distance) best predicts whether reuse is safe — and is the safe threshold task-dependent?
  • RQ3 (method, stretch): Can a position-independent, selective-recompute reuser applied to visual tokens recover accuracy for near-duplicates at a fraction of the compute?

2 months, single RTX 4090 (24 GB). Designed so a real result lands even if the stretch part doesn't.

Weeks 5–7 · Stretch

Build a similarity gate + port CacheBlend selective-recompute to visual tokens for near-duplicates. Produce the reuse-safety vs similarity curve.

Ablations, plots, short report/blog, clean repo.

8 Metrics

Quality — the differentiator (sliced, not aggregate)

  • Per-capability accuracy (OCR/text vs table vs chart vs spatial grounding)
  • Instance-flip rate — fraction of individual answers that change vs full compute
  • Hallucination rate on unanswerable questions
  • Task accuracy (step accuracy for agents, F1/ANLS for DocVQA) — for comparability

11 Related papers — read these to understand the landscape

Position-independent reuse (solves the "reflow" problem — for text)

CacheBlend: Fast LLM Serving for RAG with Cached Knowledge Fusion borrow the recipe

EuroSys 2025 · Best Paper

Reuses KV of text chunks regardless of position, then selectively recomputes the ~15% of tokens whose attention deviates most, and fixes positional encoding. 2.2–3.3× faster TTFT, no quality loss. This is the recipe you port to visual tokens.

PaperBlogCode (LMCache)

EPIC: Efficient Position-Independent Caching for Serving LLMs

ICML 2025

Formalizes "Position-Independent Caching (PIC)" and improves CacheBlend with LegoLink — recompute the first k tokens of each chunk to kill the attention-sink glitch, in linear time. Up to 8× TTFT. Read for the framing + the cheaper recompute trick.

PaperPMLR
GUI-agent KV work (fork this code)

GUI-KV: Efficient GUI Agents via KV Cache with Spatio-Temporal Awareness fork base

arXiv 2510.00536 · Salesforce AI Research

Plug-and-play (init_gui_kv(model, kv_config)), no retraining. Spatial saliency (L2 norm of hidden states) + temporal redundancy (project prior frames' keys onto current subspace). −38.9% decode FLOPs. Ships eval scripts for 5 benchmarks — your base harness.

PaperCode

STaR-KV: Spatio-Temporal Adaptive Re-weighting for KV Compression in GUI VLMs

arXiv 2606.01790

Improved successor to GUI-KV (subspace-aware scoring, temporal stability discount, entropy temperature). Source of the "76 GB for 5 screenshots" stat. Runs on UI-TARS; code public.

PaperCode
VLM / video cache reuse (the direct prior art)

VLCache: Computing 2% Vision Tokens and Reusing 98% closest prior art

arXiv 2512.12977 · SGLang

Position-agnostic reuse of the same image; formally identifies "reuse error propagation" and picks which layers to recompute. 1.2–16× TTFT. Doesn't do similar-but-different images or capability slicing — that's your wedge.

PaperCode

Sali-Cache: Dual-Signal Adaptive KV-Cache for Long-Form Video

arXiv 2602.14236

Basically your gate — an optical-flow inter-frame similarity signal; redundant frames "bypass computation entirely through cache reuse" above threshold θ. Reports aggregate accuracy only. Great to cite for "the video version exists."

Paper

InfiniPot-V: Memory-Constrained KV Compression for Streaming Video

arXiv 2506.15745

Temporal-axis Redundancy: cosine similarity of Key embeddings across frames to prune static content, under a hard memory cap. On-device streaming framing.

Paper

KVShare: Semantic-Aware KV Cache Sharing

arXiv 2503.16525

Semantic (not byte-identical) KV reuse for LLM/MLLM via alignment + differential editing, +60% hit rate. For MLLM it still needs the image identical, only text varies — leaving the image-similarity gate open.

Production baselines (exact-match reuse)

vLLM V1 — Prefix caching for multimodal models baseline

GitHub PR · production

Hashes image content (mm_hashes) into block keys; reuses KV only on byte-identical images. The exact-match baseline your approximate version aims to beat.

PR #11187

LMCache — multimodal KV reuse in vLLM V1

blog · production

Hashes vision tokens, reuses KV of repeated images across requests. Notably documents a real bug/privacy risk when placeholder matching ignores actual image identity — a hint at how fragile reuse gating is.

Reliability of compression (the "hidden damage" evidence)

Instance-level instability of quantization / token reduction

EMNLP 2025

AWQ/GPTQ W4: ~0.9–1.5% average accuracy drop but ~19–21% of predictions flip. The headline number lies — the empirical backbone of your reliability angle.

Paper

Evaluating PTQ on Reliable VQA with MLLMs

arXiv 2602.13289

PTQ degrades accuracy AND calibration on Qwen2-VL/Idefics3; a confidence "Selector" + int4 recovers most at ~75% less memory. Method template for the reliability-recovery angle.
